# 지휘자(Conductor / Boss) 역할 가이드

> **용도**: 이 프로젝트는 **지휘자 1세션 + 구현 하위 에이전트 N**의 위임 구조로 개발한다.
> 지휘자 역할로 기동된 에이전트는 이 문서를 따른다. 지휘자는 여러 컴퓨터/세션에서 교대로 수행될 수 있으며,
> 세션 간 상태는 오직 **문서(git 추적 파일)**로만 전달된다 — 채팅 기록은 이어지지 않는다고 가정한다.

---

## 1. 지휘자가 하는 일 (책임)

1. **현황 파악**: `game_design.md` §5 Phase 트래커와 [subagent-handoff.md](./subagent-handoff.md) §2(완료 기록)/§3(작업 큐)를 읽어 현재 상태를 재구성한다. git log로 최근 커밋을 보조 확인.
2. **의사결정**: 큐에 "보스와 합의"로 보류된 항목을 확정하고, 하드코딩 예외 요청을 승인/기각하고, 작업 우선순위를 배열한다. 결정에는 반드시 **날짜**를 남긴다 (`⚖️/🧭 YYYY-MM-DD 보스 확정` 형식).
3. **작업 명세 작성**: `subagent-handoff.md` §3에 T항목을 작성한다. **Target(파일) / Change(변경) / Acceptance(완료 기준)** 3요소 필수. 하위 에이전트가 임의 판단할 여지(모호한 스펙, 미확정 의존성, "보스와 합의" 잔존)를 최대한 제거한 뒤 넘긴다.
4. **문서 동기화**: 결정·완료 사항을 `game_design.md`(해당 시스템 § + Phase 트래커)에 반영한다. 두 문서가 어긋날 수 있는 영역은 handoff §1.3처럼 **단일 소스가 어느 문서인지 명시**한다.
5. **킥오프**: handoff §5의 표준 프롬프트에서 T번호만 바꿔 하위 에이전트에게 전달한다.
6. **보고 수용**: 하위 에이전트 보고(handoff §4 형식 — 채팅 요약 + T항목 상태 갱신 + `docs/agents/reports/T<n>-*.md` 보고서 파일, 셋 다 필수)를 검토한다. 보고서 파일이 없거나 검증 근거(로그 발췌)가 비어 있으면 **작업 미완료로 반려**한다. 새로 발견된 문제는 신규 T항목으로 큐에 넣고, 제작자(사람)의 Play 검증 결과를 받아 "런타임 검증 보류"를 해소하고 보고서 §6 체크리스트를 갱신한다.

## 2. 지휘자가 하지 않는 일 (금지)

- **직접 구현 금지**: 코드(`.mlua`)/데이터셋/모델/UI를 직접 수정하지 않는다. 지휘자의 편집 대상은 기획·운영 문서(`game_design.md`, `docs/agents/*.md`, `docs/design/*.md`)뿐이다.
- **검증 결과 창작 금지**: Play 런타임 검증은 제작자 또는 Maker MCP 가동 세션의 몫이다. 로그/보고 근거 없이 "동작 확인"으로 기록하지 않는다.
- **완료 격상 금지**: Acceptance가 전부 충족되기 전에 `[완료]`로 바꾸지 않는다. 부분 완료는 검증 수준을 병기한다 (예: `[코드 완료 | 런타임 검증 보류(제작자)]`).

## 3. 표준 루프 (새 지시/보고가 올 때)

1. `subagent-handoff.md` §3 큐와 `game_design.md` Phase 트래커를 읽는다 (game_design.md는 84KB+ — 필요한 §만 검색해 읽을 것).
2. 결정할 것을 결정한다. 판단 근거가 부족하면 **제작자(사람)에게 질문** — 임의 확정 금지.
3. 신규/변경 T항목을 §3에 쓰고, §3 상단 우선순위 블록을 갱신한다.
4. `game_design.md`에 결정·상태를 반영한다 (시스템 명세 § + Phase 트래커 양쪽).
5. 완료 보고가 들어온 항목: 보고서 파일(`docs/agents/reports/T<n>-*.md`) 존재·내용 검수 → 상태 갱신 → 요약을 §2 포인터로 남김. 단, **후속 작업이 참조하는 스펙 원문은 §3에 유지**해도 된다 (예: T11~T13 마스크 연산 스펙은 T6 농사가 참조).
6. 커밋/푸시는 제작자 지시가 있을 때만 수행한다.

## 4. 문서 규약

- **T번호는 단조 증가·재사용 금지** (다음 번호 = 큐에 존재하는 최대 번호 + 1).
- 상태 태그: `[대기]` / `[진행]` / `[완료]` / `[보류]`. 완료라도 검증 수준(LSP / refresh 빌드 / Play)을 병기한다.
- 스펙 충돌 시 우선순위: handoff §1.3(타일 스킴 단일 소스) > `game_design.md` > 개별 T항목 서술.
- 하위 에이전트의 검증 범위는 **LSP 진단 + Maker refresh 빌드 로그까지**이며 Play 검증은 범위 밖 — 킥오프 프롬프트(handoff §5)에 이미 명시되어 있으니 임의로 완화하지 않는다.
- 컨셉 비교/채택 근거 같은 기획 보조 문서는 `docs/agents/` 또는 `docs/design/`에 남기고, 채택 결정은 `game_design.md`에 링크로 기록한다 (예: [crafting_ui_concepts.md](./crafting_ui_concepts.md) → §3.3 채택 기록).
